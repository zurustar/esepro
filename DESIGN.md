# esepro — Design Rationale & Non-Goals

`esepro` の設計の本体は「**何を作ったか**」ではなく「**何を削ったか、そしてなぜ削って問題ないと言えるか**」にある。
「ese（似て非なる）」という名前のとおり、本プロジェクトは **RFC3261 を徹底的に削った「人間が読める範囲の最小実装」** であり、教育・実験・相互接続検証を用途とする。

とりわけ存在意義は、**対抗する SIP ソフトウェアの動作検証における「応答する相手役（test peer）」** である。検証対象は単体では動かせず、SIP メッセージを受けて何らかの反応を返す相手がいなければテストが成立しないからだ。そして検証の過程では「この応答だけ欲しい」「このエラーを注入したい」といった、その場限りの機能が必要になりうる。スクリプト言語（Python）を採用し**可読性を最優先する**のは、こうした挙動を利用者がその場でコードに書き足せるようにするためであり、これは美観ではなく **hackability という機能要件** である。

このドキュメントは、その「削り」を意図的な設計判断として明文化する。各行は次の形をとる：

> **削ったもの** → **設計思想（なぜ削ったか）** → **OK と言える前提（スコープ）** → **前提が崩れたときの失敗モード**

コードは「何を削ったか」すら語らない。まして「なぜ削って安全か」はどこにも書かれていない。したがってこの根拠表は、コードからは再導出できない、本プロジェクト唯一の一次設計情報である。

---

## 1. 適用スコープ（前提の前提）

以下の「削っても安全」は、すべて次の運用前提の下でのみ成立する。**この前提を外れた瞬間、下表の「失敗モード」が現実化する。**

- **信頼できる閉域ネットワーク**で動作する（公開インターネットに晒さない）。
- **協調的な相互接続検証**が用途（攻撃者ではなく、正しく動こうとする SIP UA が相手）。
- **対話的・短時間のセッション**で使う（長時間の無人運用・本番トラフィックは想定しない）。
- **可読性 > 機能網羅性 > 堅牢性 > 性能** の優先順位を受け入れる（可読性最優先は、利用者がテストに必要な挙動をコードに書き足せるようにするための手段＝hackability）。
- **不足機能は利用者がスクリプトを編集して補う**ことを前提とする（あらゆる検証ニーズを事前に網羅しない）。

## 2. 削りの設計根拠（Non-Goals & Rationale）

| ID | 削ったもの | 設計思想（なぜ削ったか） | OK と言える前提 | 前提が崩れたときの失敗モード |
|----|-----------|--------------------------|------------------|------------------------------|
| GAP-1 / CL-5 | トランザクション層（再送・タイマー A〜K・状態管理） | ステートレス設計が可読性の核。各メッセージを独立に処理する | 再送・信頼性は UA 側、または対話的な手動検証で吸収する | パケットロス時に再送されない。輻輳・タイムアウト制御なし |
| GAP-2 / CL-4 | 登録有効期限（Expires / expires）管理 | 期限切れ GC を持たず dict 上書きのみで最小化 | プロセス揮発（再起動でクリア）で十分。検証は短時間 | 古い登録が無期限に残り、長時間運用で陳腐化する |
| GAP-3 / CL-8 | 認証・認可（401/407、Digest） | 認証フローは可読性を大きく損なう。最小実装に不要 | 信頼できる閉域ネットワーク・実験用途のみ | 公開網ではオープンリレー化・なりすまし登録が可能 |
| GAP-4 | 登録解除（`Contact: *`）・複数 Contact・q 値 | 「1 AOR = 1 Contact、単一値上書き」が最も読みやすい | 1端末1登録の単純な検証シナリオ | マルチデバイス・優先度付きフォーキングの検証ができない |
| GAP-5 / CL-3 | SIP フル ABNF 構文の厳密検証 | 正規表現で必要部分のみを段階的に消費し、可読に保つ | 入力は概ね正当な SIP メッセージ（協調的な相手） | 不正・部分的メッセージで例外。受信ループが捕捉・ログ出力し当該パケットを破棄（プロセスは継続。CL-2 修正済み） |
| GAP-6 | 420/404/483 以外のエラー応答（その他 4xx/5xx/6xx）生成 | 検証に要る主要エラーパスのみに絞る | 検証対象が主要エラーパスに限られる | 他の異常系で適切なエラー応答を返さない |
| GAP-7 / CL-7 | TCP / TLS / SCTP、大メッセージの TCP フォールバック | UDP 一本に固定して読みやすさを維持 | テストメッセージは 64KB（0xffff）未満、UDP で足りる | 大きな INVITE（SDP 多数）が欠落。TLS 必須環境で使えない |
| GAP-8 | ループ検出・parallel / sequential forking | `Max-Forwards` によるホップ制限のみで最小化 | 単純なトポロジ、フォーキング不要 | ループは Max-Forwards 枯渇まで止まらない。並列呼び出し不可 |
| GAP-9 / CL-5 | ダイアログ・セッション状態管理（§12） | ステートレス設計の帰結。状態を一切持たない | ダイアログ追跡が不要な中継・検証用途 | ダイアログ単位の制御・課金・統計ができない |
| CL-6 | 文字コード判定（`decode('utf-8')` 固定） | コーデック判定を省いて簡潔化 | メッセージが UTF-8 でエンコードされている | 非 UTF-8 バイト列で例外。受信ループが捕捉して当該パケットを破棄（プロセスは継続。CL-2 修正済み） |
| CL-9 | 自動ユニット / 統合テスト | 実機 SIP UA との手動相互接続検証を前提とする | 変更頻度が低く、手動検証で回帰を確認できる | リグレッションの自動検知がない |
| CL-10 | 設定ファイル／環境変数による構成 | bind 先（domain / ip / port）は起動時のコマンドライン引数で指定でき、その他の設定・挙動はコード編集で変える前提（hackable-by-design であり、Python を採用した理由そのもの）。なお下記「補足」の定数は固定で正しく、外部化の対象ではない | 利用者が起動引数＋スクリプト編集で構成できる | 設定ファイルでの集中管理や無停止での設定変更には不向き |

> **補足: 正当に固定された定数（「マジックナンバー」ではない）**
> 以下のリテラルはプロトコル／トランスポートが定める唯一の正しい値であり、外部化の対象ではない。**固定で問題ない。**
> - `0xffff` — UDP データグラム長の最大値（+α）。`recvfrom` の受信バッファ長として用いる。
> - `5060` — RFC3261 / IANA が定める SIP の既定ポート。URI にポート指定がないときのフォールバックに用いる。
> - `70` — RFC3261 が推奨する `Max-Forwards` の既定値。

## 3. Known Defects（要修正）— 「なぜ問題ないか」が書けなかった項目

下記は §2 の表に入れようとして、**「削っても安全」と誠実に書けなかった**もの。書けない＝設計判断（削り）ではなく **欠陥** である、というのが本ドキュメントの判定基準である。

**現時点で未修正の欠陥はない。** 以下は修正履歴として残す。

| ID | 欠陥 | 状態 |
|----|------|------|
| ~~CL-1~~ | `420 Bad Extension` 応答パスで `handle_response`（当時の名前は `proc_response`）が `self.` なしで呼ばれ、`NameError` で失敗していた（GAP-6 と自己矛盾） | **修正済み** — `self.handle_response(resp)` に修正。Proxy-Require 付き非ACK で 420 が返ることを実機確認 |
| ~~CL-2~~ | 受信ループに例外処理がなく、不正・非UTF-8・部分的なパケット 1 発でプロセスが停止しうる（用途「相互接続検証」と矛盾） | **修正済み** — パケット処理を `handle()` に分離し受信ループで `try/except` 保護。不正パケットはログ出力して破棄し、プロセスは継続することを実機確認 |

## 4. このドキュメントの保守方針

- **機能を追加／削除したら、まずこの表を更新する**（コードより先に「何を・なぜ」を確定させる）。
- ある制限について「なぜ安全か」を書こうとして書けなければ、それは Known Defects に落とす。
- 重量級のプロセス文書（汎用 requirements 一式）は本プロジェクトの規模に対して過剰である。**価値の核はこの `DESIGN.md` に集約する**方針とする。
